# Design: Data Screen — Browse, Filter, Statistics, Export & Delete (Data-Layer First)

## Technical Approach

Data-layer-first slice of the roadmap "Data Screen". Add a `DataBrowserRepository` (separate from `RuleExecutionRepository`, which stays scoped to audit/throttle) that composes existing DAOs plus a new joined/aggregation/FTS DAO surface, then a thin `features/databrowser/` MVI screen over it. Mirror the proven Paging3 template (`NotificationDao.getFilteredPaged` / `NotificationRepositoryImpl` / `InboxViewModel`) and the `RulesScreen` share-sheet export flow. Schema grows via a clean destructive Room version bump (pre-launch, no `Migration`). No charts this phase — the trend series is computed and exposed, not rendered.

## Architecture Decisions

| Decision                  | Choice                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  | Rejected                                                                                                                                      | Rationale                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
|---------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Row model                 | New flat `DataBrowserRow` domain projection (rule name, package, app label, field name, field type, value text/number/date, source notification summary (`notificationTitle`, `notificationContent`, from the already-joined `notifications` table), createdAt, valueId, executionId)                                                                                                                                                                                                                                                                                                                                                                                                                            | Reuse `ExtractedFieldValue`                                                                                                                   | No existing entity joins execution+field+value+rule+app; a flat projection matches the `extracted_field_values ⋈ rule_fields ⋈ rule_executions ⋈ rules ⋈ notifications` join and keeps `contract` free of extraction internals (arch rule 7).                                                                                                                                                                                                                                                                                                                                                                                             |
| `value_text` search       | New FTS4 shadow entity `extracted_field_values_fts` (`@Fts4(contentEntity=…)`), synced exactly like `notifications_fts`; MATCH built via `FtsQuerySanitizer`                                                                                                                                                                                                                                                                                                                                                                                                                            | Plain `value_text` index + `LIKE '%x%'`                                                                                                       | Non-sargable LIKE table-scans; FTS4 is the established codebase precedent and Room auto-syncs the shadow table via triggers.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| Trend day-bucketing       | `strftime('%Y-%m-%d', created_at/1000, 'unixepoch', 'localtime')` GROUP BY in `@Query`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  | UTC epoch-day integer division                                                                                                                | LOCKED. Buckets by the user's local calendar day; `created_at` is epoch-millis `Long`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| Join graph                | `extracted_field_values ⋈ rule_fields ⋈ rule_executions ⋈ rules ⋈ notifications`, all as raw `@Query`s inside the new `DataBrowserDao` (no separate join-only DAO class)                                                                                                                                                                                                                                                                                                                                                                                                                | Fetching `rule_fields` separately per row / denormalizing `field_name`+`field_type` onto `extracted_field_values`                             | LOCKED. `fieldName`/`fieldType` (required on `DataBrowserRow`; also the source for the field-type filter and the field-name sort key) live only on `rule_fields` (`name`, `field_type` columns), reached via `extracted_field_values.rule_field_id → rule_fields.id` — every browse/filter/sort/aggregation/export query needs this join, not just the base browse query.                                                                                                                                                                                                                                                                 |
| Bulk delete w/ filters    | Preview resolves and returns the concrete matching ID list (`getMatchingIds(filter)`); confirm deletes by that exact ID set (`DELETE WHERE id IN (:ids)`) inside `withTransaction` — never by re-resolving the filter                                                                                                                                                                                                                                                                                                                                                                   | Re-running the filter's WHERE clause as a fresh SELECT (or DELETE) at confirm time                                                            | LOCKED. Re-filtering at confirm time lets rows inserted between preview and confirm slip into the delete; deleting by the exact previewed ID set guarantees only what was shown to the user is removed, per data-deletion spec ("only the originally-resolved ID set is deleted").                                                                                                                                                                                                                                                                                                                                                        |
| Repository boundary       | New `DataBrowserRepository`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             | Extend `RuleExecutionRepository`                                                                                                              | Keeps audit/throttle repo cohesive; new repo injects `RuleDao` (execution repo does not).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| Export                    | On-demand CSV/JSON of filtered set → cache file → `FileProvider` + `ACTION_SEND` + `createChooser` (reuse `RulesScreen` flow verbatim); new `core/data/export/DataCsvSerializer`                                                                                                                                                                                                                                                                                                                                                                                                        | Scheduled/background export                                                                                                                   | Locked scope; a single up-front ID snapshot makes the export set independent of concurrent Paging/deletion.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| Export row-cap / chunking | LOCKED. ID-set-only batching: resolve the filtered ID set once up front (`getMatchingIds(filter)`, ordered `created_at DESC, id`), then read fixed-size batches of exactly 1000 IDs at a time via `WHERE id IN (:batchOfIds)` against the pre-resolved snapshot — never `LIMIT`/`OFFSET` and never a live keyset query (`WHERE (created_at, id) < (:lastCreatedAt, :lastId) ...`) re-matched against the mutating table — writing each batch incrementally to the cache file's `OutputStream` (CSV rows, or JSON array opened/closed around the batches); never a single in-memory `List<DataBrowserRow>` read, never a hard row-count cap | `LIMIT`/`OFFSET` batches re-queried against the live table; a live keyset (`WHERE (created_at, id) < (:lastCreatedAt, :lastId) ORDER BY ... LIMIT n`) re-matching the filter against the mutating table; single in-memory `getAllMatching` read; hard row-count cap with silent truncation | Keeps memory flat regardless of dataset size (spec scenario: 50,000+ rows via streaming/chunked reads) with no silent data loss. Both `OFFSET`-based batching and a live keyset query re-evaluate the WHERE clause per batch, so a concurrent delete shifts the window and skips/duplicates rows — violating the data-export spec's "consistent snapshot" requirement; batching strictly by the pre-resolved ID snapshot (`WHERE id IN (:batchOfIds)`, mirroring the bulk-delete preview's `getMatchingIds`) makes the export immune to concurrent mutation because it never re-touches the WHERE clause. Requires temp-file-then-rename (or equivalent cleanup-on-error) so a mid-export failure never leaves a corrupt/partial file at the final share-sheet path. |
| Dry-run exclusion         | Every `DataBrowserDao` query (browse, search, aggregation/statistics, export batch, delete-ID resolution) carries a `rule_executions.was_dry_run = 0` predicate via the join to `rule_executions`                                                                                                                                                                                                                                                                                                                                                                                       | Filtering dry-run rows client-side after fetch; making it a UI-toggleable filter                                                              | LOCKED product decision. Dry-run executions (`RuleExecutionEntity.wasDryRun = true`) are test/preview runs, not real captured data, and are excluded by default from all four Data Browser surfaces (browsing, statistics, export, deletion).                                                                                                                                                                                                                                                                                                                                                                                             |

## Data Flow

    DataBrowserScreen ──(UiEvent)──▶ DataBrowserViewModel
        │  filter StateFlow ──flatMapLatest──▶ Pager(PagingSource) ──cachedIn──▶ PagingData<DataBrowserRow>
        │  stats: repo.statistics(filter) ──▶ header text (total / this-week / most-active rule)
        │  export: repo.exportRows(filter, outputStream) ──snapshot getMatchingIds once──▶ fixed-size (1000) id-set batch reads──▶ CSV rows / streamed JSON array written incrementally ──▶ temp-file-then-rename ──▶ share sheet (Effect)
        │  delete: repo.previewDelete(filter) [resolves + returns concrete ID list] ──▶ confirm ──▶ repo.deleteByIds(ids) [DELETE WHERE id IN (:ids) in tx]
        ▼
    DataBrowserRepositoryImpl (DataBrowserDao + ExtractedFieldValueDao + RuleExecutionDao + RuleDao + NotificationDao, @Dispatcher IO, withTransaction)
        ▼  Room (DataBrowserDao @Query): extracted_field_values ⋈ rule_fields ⋈ rule_executions (was_dry_run = 0) ⋈ rules ⋈ notifications  (+ extracted_field_values_fts)

## Interfaces / Contracts

```kotlin
// domain/model
data class DataBrowserRow(
    val valueId: String, val executionId: String,
    val ruleName: String, val packageName: String, val appName: String,
    val fieldName: String, val fieldType: RuleField.FieldType,
    val valueText: String?, val valueNumber: Double?, val valueDate: Long?,
    val notificationTitle: String?, val notificationContent: String?,
    val createdAt: Long,
)
data class DataBrowserFilter(
    val ruleIds: List<String> = emptyList(),
    val packageNames: List<String> = emptyList(),
    val fieldTypes: List<RuleField.FieldType> = emptyList(),
    val dateFrom: Long? = null, val dateTo: Long? = null,
    val searchQuery: String = "",
    val sort: DataSort = DataSort.DATE_DESC,
)
data class DataStatistics(
    val total: Int, val thisWeek: Int, val mostActiveRuleName: String?,
    val perRule: List<CountBucket>, val perApp: List<CountBucket>,
    val trend: List<TrendPoint>, // day-bucketed via strftime localtime
)

// domain/repository
interface DataBrowserRepository {
    fun browsePaged(filter: DataBrowserFilter): Flow<PagingData<DataBrowserRow>>
    fun searchPaged(filter: DataBrowserFilter): Flow<PagingData<DataBrowserRow>>
    suspend fun statistics(filter: DataBrowserFilter): Result<DataStatistics>
    suspend fun exportRows(filter: DataBrowserFilter, sink: OutputStream, format: ExportFormat): Result<Unit> // snapshots matching IDs once, then streams fixed-size (EXPORT_BATCH_SIZE = 1000) id-set batches directly to sink
    suspend fun previewDelete(filter: DataBrowserFilter): Result<List<String>> // resolves and returns the concrete ID set to delete; count = list.size drives the confirmation UI
    suspend fun deleteById(valueId: String): Result<Unit>
    suspend fun deleteByIds(ids: List<String>): Result<Int> // deletes exactly the previewed ID set (never re-resolves the filter); returns deleted count
}
```

## Testing Strategy

| Layer           | What to Test                                                                                                                                                                                                                                                  | Approach                                                                                           |
|-----------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------|
| Repository      | `Result` success/failure mapping, empty/zero-result, `withTransaction` bulk delete count via `previewDelete`+`deleteByIds`, insert-between-preview-and-confirm not deleted, export ID-snapshot immunity to concurrent delete, dispatcher injection            | MockK-faked `DataBrowserDao` (and other DAOs); `runTest`                                            |
| Serializer      | CSV escaping (comma/quote/newline), JSON shape, empty set                                                                                                                                                                                                     | Pure JUnit5 + Kotest                                                                               |
| ViewModel       | filter→Pager recreation (flatMapLatest), stats load, delete confirmation flow (`previewDelete` → `deleteByIds(ids)`, never re-filtering), export effect                                                                                                       | Turbine + fake repo                                                                                |

**Coverage gap (honest, intentional):** this project has no Robolectric dependency and no precedent for in-memory-Room/instrumented DAO tests (JVM-only JUnit5/Kotest/MockK/Turbine stack). The repository tests above fake `DataBrowserDao` itself, so they verify repository-level orchestration (transactions, mapping, dispatcher usage) but do **NOT** exercise the raw `@Query` SQL — the multi-table join (`extracted_field_values ⋈ rule_fields ⋈ rule_executions ⋈ rules ⋈ notifications`), the FTS4 MATCH query, the `strftime` trend bucketing, the `was_dry_run = 0` predicate, and the `getMatchingIds`/`getMatchingBatch` id-set batching are untested by automation in this change and rely on manual verification (real-device/emulator smoke testing) before merge. Adding Robolectric to close this gap is out of scope for this change.

## Migration / Rollout

Destructive Room bump `CURRENT_VERSION` 1→2 (pre-launch, no rows to preserve, per CLAUDE.md). Additive behind a new bottom-nav tab; rollback = revert branch and restore version constant. No other feature depends on the new surface.

## Open Questions

None. Export row-cap/chunking resolved (see Architecture Decisions): a one-time `getMatchingIds` snapshot, then fixed-size `EXPORT_BATCH_SIZE = 1000` id-set batches (`WHERE id IN (:batchOfIds)`) written incrementally to a temp file, renamed into place on success only.
